## 报告漏洞

我们非常重视安全问题。如果您发现安全漏洞，请按照以下步骤报告：

### 如何报告

**请勿在公开的 GitHub Issue 中报告安全漏洞。**

请通过以下方式私下报告：

1. **GitHub Security Advisory**（推荐）
   - 进入项目的 "Security" 标签页
   - 点击 "Report a vulnerability"
   - 填写安全公告表单

### 报告内容

请在报告中包含以下信息：

- **漏洞类型**：简要描述漏洞性质
- **影响范围**：受影响的组件、版本或功能
- **复现步骤**：详细的漏洞复现步骤
- **概念验证**：如有可能，提供 PoC 代码或截图
- **影响评估**：潜在的安全影响和风险等级
- **修复建议**：如有修复建议，请一并提供

### 响应时间

我们承诺按照以下时间线响应安全报告：

- **确认回复**：24 小时内
- **初步评估**：72 小时内
- **详细分析**：7 个工作日内
- **修复发布**：根据严重程度，1-30 天

### 漏洞严重程度

我们使用以下标准评估漏洞严重程度：

#### 严重（Critical）
- 远程代码执行
- 认证绕过
- 未授权访问敏感数据

#### 高危（High）
- 跨站脚本攻击（XSS）
- 权限提升
- 敏感数据泄露

#### 中危（Medium）
- 信息泄露
- 拒绝服务（DoS）
- 弱加密

#### 低危（Low）
- 配置问题
- 信息收集类漏洞

### 处理流程

1. **接收报告**：我们收到您的报告并确认
2. **漏洞验证**：安全团队验证漏洞的存在性和影响
3. **影响评估**：评估漏洞严重程度和影响范围
4. **修复开发**：开发并测试修复方案
5. **协调发布**：与报告者协调披露时间
6. **公开披露**：发布安全更新和公告

### 负责任披露

我们遵循负责任披露原则：

- 我们将在修复漏洞后公开披露
- 请在修复发布前不要公开讨论该漏洞
- 我们会在安全公告中适当感谢报告者（除非您希望保持匿名）

### 安全更新通知

要接收安全更新通知，请：

1. **Watch 本仓库** 并启用安全警报
2. **订阅 Releases** 以获取新版本通知

### 安全最佳实践

使用本项目时，我们建议遵循以下安全最佳实践：

- 始终使用最新的受支持版本
- 定期更新依赖
- 启用适当的日志记录和监控
- 实施最小权限原则
- 定期进行安全审计

### 适用范围

本安全政策适用于：

- 本 GitHub 仓库中的所有代码
- 官方发布版本

本安全政策不适用于：

- 第三方插件或扩展
- 用户配置错误
